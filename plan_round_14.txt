好，下面我把 Round 14：HTTP 与 API 设计基础 完整展开。

这一轮是路线 B 的第一步，核心不是先写很多接口，而是先把三件事讲清楚：HTTP 是什么、接口为什么这样设计、FastAPI 最小服务怎么起起来。HTTP 是 Web 上数据交换的基础，也是客户端—服务器协议；HTTP 用一组请求方法来表达这次请求想做什么；状态码用来表达请求结果，范围是 100 到 599。 

Round 14 的目标

这轮结束时，你要做到这些：你能用自己的话解释 GET、POST、PUT、PATCH、DELETE 大概分别适合什么场景；你能看懂一个最小 JSON API 的请求和响应；你能给自己的 ai_prep_tool 画出第一版接口草图；你能用 FastAPI 起一个最小服务，并看到自动生成的交互式文档。FastAPI 的 First Steps 明确展示了如何创建 FastAPI() 实例、定义 path operation，并且默认提供交互式 API 文档和 OpenAPI schema。 

这轮学什么，不学什么

这轮要学的是：HTTP 基础、请求方法、状态码、JSON 请求与响应、REST 风格的最小思路、FastAPI 最小入口、最基础的接口命名和返回结构。MDN 的 HTTP 总览、方法参考和状态码参考已经足够支撑这一层学习；FastAPI 的官方教程则足够支撑“最小可运行 API”。 

这轮先不学：鉴权、OAuth2、复杂数据库模型、异步任务队列、文件上传深水区、部署、反向代理和生产级安全。FastAPI 官方把安全单独放成后续章节，说明这些内容天然属于第二层，而不是 First Steps 的内容。 

我对你这一轮的总体建议

这一轮你不要急着把现有 CLI 全部改造成 Web 服务。更稳的顺序是：先把 HTTP 语义 理顺，再把你已经有的核心函数包一层 API。这样你以后写接口时不会只会抄代码，不知道为什么一个地方用 GET，另一个地方用 POST。HTTP 方法本身有明确语义，而且一些共同属性如 safe、idempotent、cacheable 也是围绕方法来定义的。 

本轮推荐资源

这一轮我建议你固定看四类资源。

第一类是 MDN 的 Overview of HTTP。它最适合拿来建立整体图景：HTTP 是什么、客户端和服务器在干什么、一次请求和响应大致是什么结构。 

第二类是 MDN 的 HTTP request methods 和 status codes。这两页足够支撑你把 GET、POST、PUT、PATCH、DELETE 以及 200、201、400、404、500 这类最常见状态码的直觉建立起来。MDN 还明确说明状态码分成五大类，而这些状态码定义来自 RFC 9110。 

第三类是 RFC 9110: HTTP Semantics。这一份不是让你整篇精读，而是当作“更权威的语义底座”。它明确说明状态码是三位整数，范围在 100 到 599；也系统定义了请求方法和语义。 

第四类是 FastAPI First Steps。这一份直接负责把前面学到的 HTTP 和 API 认知，落成一个真正能跑的最小服务。它同时也会让你第一次看到自动文档和 OpenAPI。 

3 周安排

第 1 周：HTTP 基础和请求/响应直觉

这一周只做一件事：把 请求是什么、响应是什么、方法和状态码是什么意思 理顺。HTTP 消息由请求和响应组成，请求方法描述这次请求的意义和期望结果，例如 GET 表示客户端想获取资源，POST 表示客户端向服务器发送数据；状态码则描述结果是成功、重定向、客户端错误还是服务器错误。 

你这一周最重要的学习结果，不是背术语，而是先形成这些判断：
	•	“取状态、取列表、取详情”通常先想 GET
	•	“提交一次处理任务”通常先想 POST
	•	“对象不存在”通常想到 404
	•	“参数不合法”通常会落到 4xx 这一类，而不是 5xx
	•	“服务端自己出错”才更接近 5xx。 

第 2 周：REST 风格最小接口设计

这一周的重点是：先为你现有项目画出第一版接口，而不是先写代码。我建议你围绕已经做出来的 ai_prep_tool，先设计这几类接口：
	•	GET /health：看服务活着没
	•	GET /runs：看历史运行记录
	•	GET /runs/{id}：看某一次运行详情
	•	POST /run：提交一次处理任务
	•	GET /summary/{id}：看某次任务的 summary。

这种设计背后的直觉是“把核心对象当资源来看待”。HTTP 方法本来就用于表达请求目的，而 FastAPI 的 path operation 也天然适合这种资源路径写法。 

这一周你还要开始建立一个简单原则：路径更像资源名，方法更像动作语义。也就是说，尽量少写成 /doRunNow 这种“把动作塞进路径”的形式，而是优先考虑资源和状态。这里我是在用 HTTP 方法语义和常见 REST 风格做工程建议；它和 MDN 对方法语义、以及 FastAPI path operation 的组织方式是一致的。 

第 3 周：FastAPI 最小服务落地

这一周才开始真正写代码。FastAPI 的最小路径非常直接：导入 FastAPI，创建 app，定义一个 @app.get(...) 路由，返回一个字典，然后运行开发服务器。FastAPI 官方教程明确说明了这条路径，并且会自动生成交互式 API 文档与 OpenAPI schema。 

我建议你这周最少只做两个接口：
	•	GET /health
	•	GET /runs

原因很简单：这两个接口最容易接你现在已有的 SQLite 和工具历史记录，而且不会一下子把复杂度拉得太高。等这两个稳定了，再做 POST /run 会更顺。这个排序是工程建议，但它是基于你当前项目形态做出的最稳安排。

这一轮最容易踩的坑

第一个坑，是还没理清 HTTP 语义，就急着堆接口。HTTP 方法不是随便选的，它们本来就有语义差异，而且还带有 safe、idempotent 等属性。 

第二个坑，是一上来就做很多 POST 接口，却不先做只读接口。对你当前项目来说，先把 GET /health 和 GET /runs 做顺，会比一开始就把“触发处理 + 文件路径 + 参数校验”全塞进一个 POST 更稳。这是工程建议，但和 FastAPI 从最简单 path operation 起步的节奏是一致的。 

第三个坑，是会写 FastAPI 路由，但不知道状态码为什么这么选。状态码本来就用于表达请求结果，而且分成明确的五大类；如果你把 404、400、500 混着用，后面接口会越来越难维护。 

第四个坑，是把 API 路径当成“随手命名的函数名”。这一轮你要开始练“资源路径 + 方法语义”的思路，而不是仅仅把 CLI 参数搬成 URL。这个建议建立在 HTTP 方法语义和 FastAPI path operation 的天然匹配上。

练习 13：接上 SQLite 历史记录

在 api.py 里新增第二个接口，先直接复用你上一轮的数据库函数：

from fastapi import FastAPI
from db import init_db, list_runs

app = FastAPI()

@app.get("/health")
async def health():
    return {"status": "ok"}

@app.get("/runs")
async def runs():
    init_db()
    rows = list_runs()
    return {"runs": rows}

这一步的重点不是“代码多高级”，而是你第一次把 已有工具能力 通过 API 暴露出来了。FastAPI 官方明确说路径操作函数在收到对应 URL 和方法的请求时会被调用，而且可以直接返回字典。  

练习 14：再补一个详情接口

继续加：

from db import get_run_by_id
from fastapi import HTTPException

@app.get("/runs/{run_id}")
async def run_detail(run_id: int):
    row = get_run_by_id(run_id)
    if row is None:
        raise HTTPException(status_code=404, detail="run not found")
    return {"run": row}

你这一步会第一次练到“用状态码表达接口结果”。404 的语义来自 HTTP 状态码体系；FastAPI 用 HTTPException 来返回这类错误也很自然。状态码参考页明确把 404 放在客户端错误类里。
