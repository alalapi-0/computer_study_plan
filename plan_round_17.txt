这轮学什么，不学什么

这轮要学的是：用 APIRouter 做多文件结构，做最小配置管理，整理日志和 API 元数据，补一个“认证概念级”入口，理解最小 CORS 场景，补部署前检查项，以及用 Uvicorn 和 Docker 把服务跑起来。FastAPI 官方文档分别对 Bigger Applications、Metadata、Security、CORS 和 Deployment 做了独立章节，说明这些正好构成了“从能跑到更像服务”的这一层能力。

我对你这一轮的总体建议

第 1 周：结构化服务，把单文件 API 拆开

我建议你这一周至少把这些配置抽出来：
	•	app_name
	•	api_version
	•	debug
	•	database_path
	•	log_level

FastAPI 的 Metadata and Docs URLs 章节明确说，你可以自定义 API 的 title、summary、description、version，也可以自定义 OpenAPI URL 和 docs URLs。也就是说，这一周你可以让 /docs 不再是“默认壳子”，而是开始像你自己的服务。 

日志这边，这一轮你不需要引入新框架，但至少要做到：主入口统一初始化日志；路由里不随手 print()；错误路径和成功路径都有清楚日志。Uvicorn 的 settings 和 deployment 文档都显示，服务运行本身就有自己的 host、port、log-level 这类配置项，这也意味着“服务运行”和“应用日志”最好都开始有统一入口。 

第 3 周：最小认证概念、部署前检查项、Docker 运行 API

这一周主题是：让服务更像“准备对外暴露”的状态。

先说最小认证概念。FastAPI 的 Security - First Steps 不是让你马上做完整登录系统，而是先让你理解：OAuth2 Password flow + Bearer token 是很常见的一条路；OAuth2PasswordBearer 需要一个 tokenUrl，客户端会把用户名和密码发到那个地址换 token，然后后续请求带着 token 来访问受保护的接口。FastAPI 还特别说明 Bearer token 往往是大多数用例的合适起点。 

这一轮我建议你先不真正做完整 token 颁发，而是只做两件事：第一，理解“哪些接口未来可能需要保护”，例如 POST /run；第二，给自己留一个简单的认证扩展点。这样你就不会在下一轮之前把安全完全忽略掉，但也不会一下子跳进认证深水区。这个建议是基于 FastAPI Security First Steps 的定位做出的。 

然后是部署前检查项。FastAPI 的 Deployment Concepts 章节把上线前要考虑的事总结得很清楚：HTTPS、启动方式、自动重启、进程副本数、内存、以及“启动前需要做的事”。你这一周至少要给自己写一份 checklist，里面包括：
	•	配置是否都能从环境变量读
	•	数据库路径是否正确
	•	docs 和 metadata 是否清楚
	•	错误接口返回是否合理
	•	Docker 能否正常跑起来
	•	端口是否正确发布

这些检查项不是“可有可无”，而是服务从本地实验往外迈一步前必须想清楚的东西。 

最后是 Docker 运行 API。Uvicorn 的 settings 文档明确给出了最小命令，比如 uvicorn main:app --host 0.0.0.0 --port 8000；Docker 官方则明确说明，要让容器端口对外可见，需要 -p/--publish。这意味着你这一轮最好至少能做到：
	•	容器里启动 FastAPI / Uvicorn
	•	用 docker run -p 8000:8000 ... 把端口映射出来
	•	在宿主机访问服务和 /docs

Docker 文档还特别说明，默认桥接网络下容器端口不会自动对外可见，必须 publish。 

这轮建议的最小运行方式

我建议你这一轮的服务入口先统一到类似：

uvicorn api.main:app --host 0.0.0.0 --port 8000

Uvicorn 官方 settings 文档直接给出了这种配置方式，而且说明命令行、程序化调用和环境变量都是可选配置方式；同时它还提醒，如果你用 reload=True 或 workers=NUM 的程序化方式，应该放在 if __name__ == "__main__" 保护下。 

如果你用 Docker 跑，最小命令可以是类似：

docker run --rm -p 8000:8000 ai-prep-tool:demo

端口发布这一点，Docker 官方文档说得很明确。

这一轮最容易踩的坑

第一个坑，是只把文件拆开，却没有真正按职责拆。APIRouter 的意义不是“多几个文件”，而是按资源或领域分组路径操作。FastAPI 官方 Bigger Applications 讲的就是这一点。 

第二个坑，是继续把配置硬编码在代码里。FastAPI 的 Settings and Environment Variables 已经明确展示了把设置放进单独模块并通过环境变量覆盖的方式；Pydantic Settings 也明确说它对生产里的 secrets 和环境变量很友好。 

第三个坑，是一提安全就直接冲向完整 JWT 系统。FastAPI 的 Security - First Steps 更适合你现在这一步：先理解 Bearer token 和 tokenUrl 的角色，先知道服务哪些端点未来需要保护。 

第四个坑，是 Docker 里服务起了，但端口没 publish，结果宿主机访问不到。Docker 官方对 -p/--publish 的说明已经足够明确了。 

第五个坑，是直接把 uvicorn --reload 当作“部署方式”。Uvicorn 的 deployment 文档明确区分了本地开发和生产部署：本地开发常用 uvicorn --reload，而生产会考虑 worker、反向代理等。你这一轮先理解这个边界就够了。

from functools import lru_cache
from pydantic_settings import BaseSettings

class Settings(BaseSettings):

	•	运行时端口是否明确
	•	启动命令是否清楚

FastAPI 的 Deployment Concepts 就是在提醒你：上线前需要想清楚的不只是“怎么启动”。  

练习 14：用 Uvicorn 直接启动服务

先本地直接跑一次：

uvicorn api.main:app --host 0.0.0.0 --port 8000

Uvicorn 官方 settings 文档明确说明了 host、port 这些参数。  

练习 15：调整 Dockerfile 让它跑服务

把 Dockerfile 默认命令改成更贴近服务运行，例如：

CMD ["uvicorn", "api.main:app", "--host", "0.0.0.0", "--port", "8000"]

然后 build：

docker build -t ai-prep-tool:demo .

练习 16：发布端口并运行容器

运行：

docker run --rm -p 8000:8000 ai-prep-tool:demo

然后在宿主机浏览器访问：

http://127.0.0.1:8000/docs

Docker 官方明确说明，必须 publish 端口，否则容器里的服务默认不能直接从宿主机访问。  

练习 17：确认服务在容器里也能工作

至少检查：
	•	/health
	•	/runs
	•	/docs

能打开、能返回、能看文档，就说明你这一轮已经把“服务化收口”的最小闭环跑起来了。

第 3 周周末小任务

这一周结束时，你至少要做到：
	•	有一份最小安全说明
	•	有一份部署前检查清单
	•	能用 Uvicorn 启动服务
	•	能用 Docker 运行服务并通过 -p 暴露端口
	•	宿主机能访问到容器里的 /docs
