# Master Study Roadmap · 总学习路线（v1.0）

> 本文档是仓库 `computer_study_plan` 的**总目标路线图**。
> 所有阶段计划、知识点映射、周计划模板、进度规则与错题系统都挂在此文档之下。
> 任何新增学习计划都必须先回到本文档对齐目标，再决定挂到哪条主线。

---

## 1. 总目标

### 1.1 短期目标（0–18 个月）

**软考中级，高分 / 满分导向通过**，默认优先方向：

- **软件设计师**（主线，本仓库默认）

兼容备选（不作为当前主线，仅保留路径与素材库结构）：

- 数据库系统工程师
- 网络工程师
- 信息系统项目管理师

> 软考考试大纲与题型由中国计算机技术职业资格网（rkb.ruankao.org.cn）发布。
> **本仓库不缓存具体考题或考点，所有专项内容必须以最新官方大纲为准。**

### 1.2 中期目标（12–30 个月）

**补齐非科班的计算机基础短板**，建立可长期维护的知识体系。重点模块：

- 计算机组成基础
- 操作系统
- 数据结构与算法
- 数据库
- 计算机网络
- 软件工程
- 面向对象与 UML

中期目标可以由软考主线"顺带覆盖"，但深度不止步于"考过"，要追到能写代码、能解释、能向人讲清楚的程度。

### 1.3 长期目标（24–48 个月）

**跨专业考研准备**，目标方向：

- **0854 电子信息**（专业学位大类）
- 085410 人工智能
- 085404 计算机技术
- 085405 软件工程

长期考试基础：

- **数学二**（高等数学 + 线性代数）
- **408 计算机学科专业基础综合**（兼容；即使部分院校不考 408，408 也是检验计算机基础的最佳标尺）

> 院校与具体考试科目以**目标学校研究生院/学院官网当年招生目录**为准，详见 `docs/GRADUATE_SCHOOL_TRACKER.md`。
> **本仓库不基于经验贴或过往年份编造院校招生信息。**

### 1.4 更长期愿景

形成一条从「Linux/计算机基础学习」→「软考中级」→「0854/数学二/408」→「AI 工程作品集」的连续路线，**避免重复学习、碎片学习和文档膨胀**。

## 2. 三条主线（实际四条编号 lane）

仓库进度系统以 `lane` 字段标识主线归属。共 4 个 lane：

| lane 代码 | 中文名 | 短期 / 中期 / 长期定位 |
|---|---|---|
| `engineering` | 工程实操线 | 中长期主力。承接 Linux/Shell/Git/Python/工程化/服务化/AI 工程；服务于实际项目与作品集 |
| `soft_exam` | 软考中级线 | 短期主力。默认软件设计师，高分/满分导向 |
| `math2` | 数学二线 | 长期低强度主力。高数 + 线代，**不被软考挤掉** |
| `cs408` | 408 / 0854 线 | 中长期主力。在软考基础上深化到 408 层级，对接 0854 考研 |

> 在本文档之外，所有具体学习计划目录均按这四个 lane 之一归位（详见 `plans/` 下的子目录）。

### 2.1 主线 A：工程实操线（engineering）

- Linux 基础（命令行、文件系统、用户权限、进程）
- Shell 编程（变量、条件、循环、管道、重定向）
- Git / GitHub（分支、合并、PR、协作）
- 网络基础（TCP/IP、HTTP/HTTPS、DNS、SSH）
- 服务器 / VPS（远程登录、目录权限、tmux、最小服务部署）
- 自动化脚本（cron、批处理、日志轮转）
- Python 工程化（模块、包、测试、虚拟环境、打包）
- API 服务化（HTTP、FastAPI、数据层、路由）
- AI 工程素养（NumPy/pandas、sklearn、PyTorch、NLP 最小闭环）

> 这条主线就是仓库原有的 Round 00–21 + VPS 实操支线，**未被废弃**，定位为"工程实操素材库"。

### 2.2 主线 B：软考中级线（soft_exam）

按软考软件设计师考试大纲建议覆盖的核心知识域：

- 计算机组成基础（数制、运算、指令系统、寻址）
- 操作系统（进程、内存、文件系统、I/O）
- 数据结构与算法
- 数据库（关系模型、SQL、范式、设计、事务）
- 计算机网络（OSI/TCP-IP、HTTP、网络设备、协议）
- 软件工程（生命周期、过程模型、需求分析、测试）
- 面向对象分析与设计
- UML（用例图、类图、序列图、状态图、活动图）
- 信息安全基础（加密、认证、安全协议）
- 标准化与知识产权
- 程序设计语言基础（C 语言为主，Java/Python 旁参考）
- 历年真题与错题本（最高优先级，详见 `docs/ERROR_REVIEW_SYSTEM.md`）

> 具体考点以**最新官方大纲**为准，本仓库只承担"知识体系骨架 + 学习节奏 + 错题回流"。

### 2.3 主线 C：数学二线（math2）

数学二考试范围：

- **高等数学**：函数与极限、连续性、一元微分、一元积分、多元微分、二重积分、微分方程、级数（按当年大纲为准）
- **线性代数**：矩阵、向量空间基础、线性方程组、特征值与特征向量、相似矩阵、二次型

数学二**不包含**概率论与数理统计（这是数学一 / 三的范围；本仓库默认数学二）。

设计原则：

- **早期低投入、持续不断线**（每天 / 每周固定一点点）
- 不在软考前冲刺数学；软考考完后逐步加码
- 错题与计算量并重，不只看视频

### 2.4 主线 D：408 / 0854 线（cs408）

408 计算机学科专业基础综合 4 个科目：

- 数据结构（含算法）
- 计算机组成原理
- 操作系统
- 计算机网络

设计原则：

- 软考层级是入门和广度；408 需要计算题、推理与系统理解的深度
- 在软考主线完成后，复用其知识体系，做"深度加挖"
- 即使目标院校最终不考 408，408 也是检验计算机基础的最佳尺子，**所有内容兼容 408**

### 2.5 编程基础（横跨四条 lane）

- 必须至少有一门**主语言**：建议 C 语言 + Python；Java 视目标院校机试要求而定
- C 语言：服务于软考 + 408
- Python：服务于工程实操线 + AI 项目
- Java：可选；如复试机试需要再补

### 2.6 复试与求职准备（横跨）

- 项目介绍能力（作品集）
- 机试能力（数据结构 / 算法 / C/C++）
- 专业问答（操作系统、计组、网络、数据库、AI 基础）
- 详细见 `docs/PROJECT_PORTFOLIO_TRACK.md`

## 3. 目标之间的耦合关系

本节非常重要：避免重复学习与碎片学习的关键，是看清"同一个知识点在不同目标下扮演什么角色"。

### 3.1 Linux 学习如何服务于工程实践

- 让"环境搭建 / 部署 / 调试 / 日志查看"成为肌肉记忆，而不是每次都要查文档
- VPS 实操支线（`rounds/stage_03_vps_remote_ops/`）让远程工作具备最小闭环
- AI 工程项目的训练 / 推理 / 数据采集都需要 Linux 习惯

### 3.2 软件设计师如何为 408 打基础

| 知识域 | 软考层级 | 408 层级 | 关系 |
|---|---|---|---|
| 操作系统 | 概念 + 简单计算 | 调度算法、内存管理、文件系统的计算题 | 软考通过 = 408 入门 |
| 数据结构 | 常见结构 + 代码片段 | 复杂应用题（树/图/哈希/排序综合） | 软考过关后必须加码 |
| 计算机网络 | 协议层次 + 协议功能 | 详细协议字段、滑动窗口、子网计算 | 软考通过 = 408 入门 |
| 计组 | 数制 + 指令系统概念 | CPU 流水线、cache 替换、指令编码计算题 | 软考几乎没覆盖到 408 深度 |
| 软件工程 | 软考核心 | 408 不考 | 软考独有 |
| 信息安全 | 软考独有 | 408 不考 | 软考独有 |

> 结论：软考是 408 的**广度版**，而 408 是**深度版**。两者复用大纲但训练强度不同。

### 3.3 数据结构 / 操作系统 / 网络 / 数据库 在三条考试线的重叠

这四个模块是"**复用率最高**"的，本仓库的设计原则是：

- 不为每个目标各写一份大纲
- 同一份大纲下分三个梯度：「软考标准」「408 加码」「数学二无关」
- 错题统一归类到模块，但标注"软考 / 408" 来源

详见 `docs/KNOWLEDGE_MAPPING.md` 的 20 个模块映射表。

### 3.4 数学二为什么需要单独长期推进

- 数学二与软考、408 都没有强重叠；属于**独立长期项目**
- 在软考冲刺期会被自然挤压，所以必须早启动、低强度、不断线
- 数学是考研最大变量，迟启动会很被动

### 3.5 软考不是考研替代品

- 软考可以**作为"我的计算机基础是否过线"的阶段性验证**
- 但软考通过 ≠ 408 通过；408 计算题、推理题、综合题的强度高得多
- 软考通过也 ≠ 0854 复试通过；复试要项目、机试、口头表达

### 3.6 作品集如何服务于复试 / 求职 / 个人能力证明

详细项目方向见 `docs/PROJECT_PORTFOLIO_TRACK.md`，原则：

- 每个项目必须有**最小可运行版本**
- 每个项目必须标注**对应知识点 / 软考模块 / 408 模块 / 求职复试价值**
- 一个项目能服务多个目标 → 高 ROI，优先

## 4. 学习原则（硬约束）

下面这些原则在本仓库内是"硬约束"。任何新增学习计划、任何 Round、任何阶段在落地前都必须能回答这些原则的问题。

### 4.1 不追求只刷证

- 证书是结果，知识体系才是目的
- 用证书倒逼知识体系，而不是用刷题代替学习
- 即使软考不考的内容（如 408 深度），只要属于体系也要补

### 4.2 不用碎片化收藏代替学习

- 不收藏长视频清单代替练习
- 不收藏长文章代替读完它
- 每个资料必须有"是否读完 / 是否做完练习"的状态
- 资料索引在 `plans/<scope>/resources.md`（按需建立）

### 4.3 每个阶段必须有可检查产物

- 文档 / 笔记 / 错题 / 代码 / 项目 / 通过的练习 都算产物
- "感觉懂了"不算
- 阶段产物列在 `docs/STAGE_PLAN.md`

### 4.4 每周必须有低负担复盘

- 复盘不是写长篇大论，而是回答 4–6 个固定问题
- 模板在 `docs/WEEKLY_EXECUTION_TEMPLATE.md`
- 写完归档到 `records/weekly_reviews/YYYY-WW.md`

### 4.5 所有知识尽量落到代码 / 笔记 / 题目 / 项目

- 软考知识 → 错题本 + 概念笔记
- 工程知识 → 可运行脚本 + README
- 数学知识 → 错题本 + 关键例题
- 408 知识 → 错题本 + 推导笔记 + 代码实现（数据结构必做）

### 4.6 不破坏已有学习记录

- `records/` 下任何已写入的真实学习记录默认禁止删除
- 旧 Round 00 已完成状态默认禁止重置
- `progress.json` 中的 `done` / `done_at` 字段必须能被 `mark_done.sh` 长期识别

### 4.7 不基于经验贴或过期年份编造考试信息

- 软考大纲、考研大纲、招生目录必须以**最新官方来源**为准
- 本仓库内涉及具体年份的信息必须标注"截至 YYYY-MM 的官方信息"
- 没有联网或没有最新官方信息时，**只建结构和待填字段**

## 7. 维护规则

- 本文档只描述**目标与原则**。具体节奏、具体题目、具体院校信息不写在这里。
