# Decisions

本文件记录影响项目长期推进方式的架构决策。每条 ADR 都应区分“当前阶段决策”和“未来可能演进”，避免把长期方向误写成当前事实。

## ADR-0001: 当前阶段继续使用 JSON 作为状态存储

- 日期：2026-05-05
- 状态：Accepted
- 背景：当前项目已有 `progress.json` 作为状态单一事实源，并由 Web UI / `mark_done.sh` 写入、`progress_data.js` 镜像到页面。
- 决策：当前阶段继续使用 JSON / JSONL 作为任务状态、动作记录、任务反馈和本地存档的存储格式。
- 原因：JSON 易读、易版本管理，配合轻量本地服务即可支持当前单人学习工作流。
- 后果：短期内无需引入数据库；当事件日志和查询需求明显超过 JSON 能力时，再评估 SQLite 或后端服务。

## ADR-0002: 当前阶段保留静态 progress.html，不立即引入前端框架

- 日期：2026-05-05
- 状态：Accepted
- 背景：当前 `progress.html` + `progress_ui.js` 已承担学习工作区、内联教程、工程终端、记录弹窗、练习脚本运行、存档和复盘入口。
- 决策：当前阶段保留原生 HTML + JavaScript + 轻量本地服务的实现方式，不立即引入 React、Vue 或其他前端框架。
- 原因：项目首先需要统一数据模型和任务体系，过早引入框架会增加维护成本和学习门槛。
- 后果：继续控制前端复杂度；当原生页面难以承载更复杂的任务操作、历史记录查询和反馈展示时，再评估前端架构升级。

## ADR-0003: 当前阶段保留 mark_done.sh，不立即改成完整后端服务

- 日期：2026-05-05
- 状态：Accepted
- 背景：Web UI 已成为正式学习入口；`mark_done.sh` 仍提供任务标记、撤销、状态查看和生成 `progress_data.js` 的兼容闭环。
- 决策：当前阶段继续保留并兼容 `mark_done.sh`，复杂交互优先通过 `scripts/progress_server.py` 提供本地 API。
- 原因：shell + 内嵌 Python 的方式对本地学习足够轻量，也能保护 Round 00 现有可运行能力。
- 后果：脚本复杂度需要被控制；如果后续读写逻辑明显变复杂，再拆出 Python 脚本或 API 服务。

## ADR-0004: 默认直接推进 main

- 日期：2026-05-05
- 更新：2026-07-06
- 状态：Accepted（替代原“必须独立分支”规则）
- 背景：项目会长期由多轮小任务推进，需要保证每轮可验证、可提交、可回滚。
- 决策：默认直接在 `main` 上完成、验证、commit，并在 remote 存在时 push `main`。独立分支只在用户明确要求 PR 审查、需要实验性大改、或 `main` 推送失败时使用。
- 原因：本仓库是单人本地学习总控仓库，直接主线推进更符合当前协作节奏；PR 流程会制造额外文档和操作成本。
- 后果：每轮必须更重视本地验证、提交范围说明和最终汇报；push 失败、冲突或验证连续失败时立即停止。

## ADR-0005: 动作记录与反馈先保持本地 JSON/JSONL

- 日期：2026-05-05
- 状态：Accepted
- 更新：2026-07-06
- 背景：Web UI 已经具备“用户动作 -> 记录 -> 反馈”的本地闭环，覆盖 Round 00–21 和计划阅读入口。
- 决策：继续使用本地 JSON/JSONL 保存进度、动作日志、任务反馈和存档；暂不引入数据库或完整后端架构。
- 原因：当前单人学习仓库更需要稳定、可读、可版本管理的数据文件，不需要复杂服务化。
- 后果：短期不会做登录、云同步或多人系统；服务化/API 化只在静态文件和本地脚本无法支撑需求时启动。

## ADR-0006: 当前阶段只保留一门 Linux 课程

- 日期：2026-07-18
- 状态：Accepted
- 背景：仓库曾同时承载软考、考研、Python/算法/后端/ML 等多条学习路线，导致产品定位与内容边界混乱。
- 决策：进入单课程验证阶段，只保留并推进 `linux-foundations`（Linux 基础与工程实践）；当前正式产品范围不含其他课程。
- 原因：需要先用一门课验证持续学习、记录、反馈、通关感与网页体验是否成立。
- 后果：进度系统收敛为单一 lane；新功能必须优先服务 Linux 垂直切片；多课程仅作为长期开放能力保留在模型层。

## ADR-0007: 删除非 Linux 正式课程内容

- 日期：2026-07-18
- 状态：Accepted
- 背景：软考 / 数学二 / 408 计划与 Round 03–05、07–21 等非 Linux 路线使仓库目标发散。
- 决策：按 `docs/CONTENT_AUDIT.md` 与 `docs/REMOVAL_MANIFEST.md`，将分类为 `REMOVE_NON_LINUX` 的内容移出当前工作树。
- 原因：单课程验证需要清晰边界；混杂内容会污染进度、文档与用户心智。
- 后果：工作树仅保留 Linux 正式课程与平台；恢复依赖备份标签与 Git 历史，不在工作树内另建归档目录。

## ADR-0008: 平台模型保持可扩展，但当前禁止第二门课程

- 日期：2026-07-18
- 状态：Accepted
- 背景：长期可能支持多课程，但当前尚无验证闭环。
- 决策：底层保持 course-agnostic（Course/Module/Lesson/Task 等），但本阶段禁止注册或推进第二门正式课程。
- 原因：避免为不存在的多课程需求过度设计，同时保留将来扩展空间。
- 后果：可新增通用平台能力；不得把非 Linux 内容重新加入 active content，直至 ROADMAP Phase 10 条件满足。

## ADR-0009: 先完成 Linux Round 00 垂直切片，再扩展其他 Linux 模块

- 日期：2026-07-18
- 状态：Accepted
- 背景：Linux 材料已含 Round 00/01/02/06 与 VPS 支线，但游戏化闭环尚未打通。
- 决策：优先把 Round 00 做成可验证的网页交互垂直切片，再完善其他 Linux 模块。
- 原因：垂直切片能最快验证记录、反馈、通关与坚持假设。
- 后果：短期内不以“铺满全部 Linux 模块内容”为优先；Round 00 可运行性为硬约束。

## ADR-0010: UI/UX 在下一阶段单独重新设计

- 日期：2026-07-18
- 状态：Accepted
- 背景：本轮目标是清理、收敛与重新定位，不是视觉改版。
- 决策：UI/UX 系统重设计放在 ROADMAP Phase 6 / 下一轮单独进行；本阶段仅允许单课程文案与 lane 兼容调整。
- 原因：先固定产品范围与内容真源，再投入视觉与信息架构成本。
- 后果：`progress.html` 可能保留历史 CSS/DOM 残留；不得在本阶段引入前端框架或用设计稿无审查覆盖。

## ADR-0011: 被删除内容通过 Git 历史保留，不在当前工作树中归档

- 日期：2026-07-18
- 状态：Accepted
- 背景：删除非 Linux 内容需要可恢复，又不能继续污染当前产品树。
- 决策：删除仅作用于工作树；以备份标签 `pre-linux-only-refactor-20260718-1029` 与 Git 历史作为恢复源，不另建 `archive/` 镜像树。
- 原因：双份归档会造成“假删”与引用漂移。
- 后果：恢复使用 `git restore --source=<backup-tag> <path>`；文档中的历史提及不等于正式范围。
