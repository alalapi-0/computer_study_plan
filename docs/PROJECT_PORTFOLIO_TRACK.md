## 1. 文档定位

本仓库当前不存放真实项目代码（按 `docs/governance/repo_rules.md` 的约束，真实项目代码应独立成项目仓库）。
但本仓库需要**追踪项目状态、对应知识点、对应考点、对应复试价值**，以便：

- 让学习与作品集互相回流（学到的→做项目；做项目时发现的→反馈到学习计划）
- 在复试 / 求职时一眼看清自己的项目清单
- 避免"学了一堆但做不出任何东西"

## 4. 作品集运作原则

### 4.1 一个项目能覆盖多个目标，优先做

如果一个项目同时对应"软考数据结构 + 408 数据结构 + AI 数据质量经验"，那么它的 ROI 是 3×，优先级最高。

### 4.2 每个项目必须有 MVP

定义 MVP 是为了避免"项目永远在构思"。MVP 三要素：

1. **输入**：能从命令行 / API / 文件传入数据
2. **处理**：至少有 1 个核心功能跑通
3. **输出**：能写到文件 / 控制台 / API 响应

### 4.3 项目代码不放本仓库

本仓库只放项目索引（即本文件 + 项目卡片）。代码放独立 GitHub 仓库。
原因：
- 避免本仓库膨胀
- 项目代码有自己的迭代节奏，不应与学习仓库混在一起
- 项目仓库的 README 是求职 / 复试时直接展示的

### 4.4 1 页项目介绍

每个完成 MVP 的项目，应能写出**1 页项目介绍**（500–800 字），结构：

1. 项目背景（为什么做）
2. 核心功能（做了什么）
3. 关键技术（用了什么）
4. 难点与解决方案
5. 项目链接 / 演示

这 1 页就是复试展示用的"作品集卡片"。

## 5. 与学习计划的回流

| 学习中遇到的情况 | 项目侧动作 |
|---|---|
| 学到一个新数据结构 | 看是否能用在现有项目里 |
| 学到一个新 API / 工具 | 看是否能升级现有项目 MVP |
| 错题本里反复错某个模块 | 启动一个小项目把概念落到代码 |
| 复试模拟时讲不清楚某个项目 | 回到项目仓库补 README 与文档 |
