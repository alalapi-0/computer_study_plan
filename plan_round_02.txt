这一轮学什么

这一轮只学到“能用”这一层：
	•	echo
	•	>
	•	>>
	•	|
	•	grep
	•	wc
	•	sort
	•	uniq
	•	一个非常小的 shell 脚本

Bash 官方文档把 redirection 定义为把命令的输入输出改到别的地方，把 pipeline 定义为用 | 或 |& 连接的一串命令，前一个命令的输出会接到下一个命令的输入上。Missing Semester 的 shell tools 讲义也强调：shell scripting 的优势就在于创建命令流水线、把结果写入文件、从标准输入读数据这些事情非常自然。 

这一轮先不学什么

	•	if / for / while
	•	sed / awk

	•	Git 分支冲突
	•	rebase
	•	stash
	•	远程仓库协作细节

这不是因为它们不重要，而是因为你现在最需要的，是先把 “命令行自动化感” 和 “版本记录感” 建起来。Missing Semester 也专门提醒过：bash 很适合短小、简单、一次性的脚本，但更复杂的脚本更推荐用 Python 之类更成熟的脚本语言，这和你“以后要接 AI 项目”的目标是吻合的。

二、这 3 周的安排

这轮还是按 3 周、每周约 8 小时 来走。

第 1 周：重定向和管道

本周目标

你要真正理解三件事：
	•	>：把输出写进文件
	•	>>：把输出追加到文件末尾
	•	|：把左边命令的输出接给右边命令

Yale 的 Linux 入门材料讲得很清楚：> 会把输出重定向到文件，不存在就创建，存在就覆盖；| 是把左边命令的输出作为右边命令的输入；>> 是追加；而且 uniq 只会去掉相邻重复项，所以常见搭配是 sort | uniq。 

本周只练这些命令

echo
cat
>
>>
|
grep
wc
sort
uniq

第 1 周练习目录

mkdir -p ~/cli-lab/round2/week1
cd ~/cli-lab/round2/week1
pwd
ls

练习 1：认识 > 和 >>

echo "apple" > fruits.txt
cat fruits.txt

echo "banana" >> fruits.txt
cat fruits.txt

echo "orange" >> fruits.txt
cat fruits.txt

你现在要体会到：
	•	第一次 > 是创建并写入
	•	后面的 >> 是继续往后加
	•	cat 是把内容直接打印出来

这些行为和 Bash 手册、Yale 的 pipe/filter 教程是一致的。 

练习 2：感受覆盖

echo "new line" > fruits.txt
cat fruits.txt

现在你会看到前面的内容没了，只剩一行。
这一步很重要，因为以后写日志和生成中间文件时，很多人最容易在这里误覆盖。Yale 的教程也专门提醒了 > 会静默覆盖已存在文件。 

练习 3：把几个命令串起来

echo "cat" > animals.txt
echo "dog" >> animals.txt
echo "cat" >> animals.txt
echo "bird" >> animals.txt
echo "dog" >> animals.txt
echo "cat" >> animals.txt

cat animals.txt
sort animals.txt
sort animals.txt | uniq
sort animals.txt | uniq | wc -l

这里你要体会的是：
	•	sort 先把相同内容排到一起
	•	uniq 去掉相邻重复
	•	wc -l 统计行数

Yale 的示例正是这种“把简单工具用管道串起来”的思路；它还明确解释了 uniq 为什么只处理相邻重复，以及为什么常搭配 sort。 

练习 4：过滤

echo "error: login failed" > app.log
echo "info: job started" >> app.log
echo "error: file missing" >> app.log
echo "info: job finished" >> app.log

cat app.log
grep "error" app.log
cat app.log | grep "error"
cat app.log | grep "info" | wc -l

这一步你先别追求正则表达式，只要把 grep 理解成“筛出包含某段文字的行”就够了。

第 1 周小成果

做一个 label_stats.txt：

echo "ok" > labels.txt
echo "blur" >> labels.txt
echo "ok" >> labels.txt
echo "bad" >> labels.txt
echo "blur" >> labels.txt
echo "ok" >> labels.txt

sort labels.txt | uniq > label_stats.txt
cat label_stats.txt

这个小成果已经开始接近 AI 数据项目里的“粗统计”和“预检查”了。

第 2 周：最小 shell 脚本和命令组合感

本周目标

这一周不追求 Bash 语法大全，只做两件事：
	•	会写一个最简单的 .sh 文件
	•	会让脚本帮你串一组固定命令

Missing Semester 的 shell tools 讲义明确说，shell script 是在 shell 命令和管道基础上的下一步；它擅长做 shell 相关任务，比如创建命令流水线、保存结果、读标准输入。 

练习 5：你的第一个脚本

先建目录：

mkdir -p ~/cli-lab/round2/week2
cd ~/cli-lab/round2/week2

创建脚本文件：

touch count_errors.sh

把下面内容写进去。你可以用文本编辑器打开写，也可以直接复制粘贴。

#!/bin/bash
cat app.log | grep "error" | wc -l > error_count.txt
cat error_count.txt

然后准备测试文件：

echo "error: login failed" > app.log
echo "info: job started" >> app.log
echo "error: file missing" >> app.log
echo "info: job finished" >> app.log

执行脚本：

bash count_errors.sh

你只要能跑通，就够了。
这一轮先不要求你理解 shebang、权限、参数、变量那些细节。

练习 6：再做一个“标签统计脚本”

新建文件：

touch count_labels.sh

内容写成：

#!/bin/bash
sort labels.txt | uniq > unique_labels.txt
cat unique_labels.txt

再准备数据：

echo "ok" > labels.txt
echo "blur" >> labels.txt
echo "ok" >> labels.txt
echo "bad" >> labels.txt
echo "blur" >> labels.txt
echo "ok" >> labels.txt

执行：

bash count_labels.sh

第 2 周的关键理解

这一周你只要形成这个感觉就行：

脚本不是高深程序，它只是把你本来要手敲的一串命令，保存成一个可重复执行的文件。

这正是 bash 在简单任务上最有价值的地方。Missing Semester 也明确建议：bash 适合短小、简单、一次性的脚本；再复杂就切 Python。 

第 2 周小成果

第 3 周：Git 最小工作流

本周目标

你这周只建立一个闭环：

初始化仓库 → 看状态 → 添加文件 → 提交 → 看历史

GitHub 官方文档要求最小起步动作是安装 Git 并设置用户名和邮箱；Pro Git 的前两章和 GitHub Quickstart 也都把“建仓库、记录改动、查看历史”列为最前面的基础内容。 

先检查 Git

git --version

如果没装，再按 GitHub Docs 的 Set up Git 走。装完后配置身份：

git config --global user.name "你的名字"
git config --global user.email "你的邮箱"

这两个步骤正是 GitHub 官方 setup 文档列出的最小配置。 

Git 练习目录

mkdir -p ~/cli-lab/round2/week3
cd ~/cli-lab/round2/week3
mkdir git_lab
cd git_lab

练习 7：初始化仓库

git init
git status

Git 是版本控制系统，Missing Semester 把它定义为跟踪文件夹及其内容变化的一系列“快照”的工具；即使一个人工作，它也能让你回看旧版本、记录每次改动原因。 

练习 8：第一次提交

echo "# round2 git lab" > README.md
git status
git add README.md
git status
git commit -m "init repository"
git log --oneline

你要观察三个状态变化：
	•	创建后是 untracked
	•	git add 后进入 staged
	•	git commit 后进入历史

GitHub Docs 的 quickstart 示例也使用了同样的最小流：git status，然后 git add README.md && git commit -m "Add README"。 

练习 9：第二次提交

echo "this repo is for round2 practice" >> README.md
git status
git diff
git add README.md
git commit -m "update readme"
git log --oneline

这里你会第一次看到：
	•	文件改了但还没提交
	•	git diff 能看当前改动
	•	git log --oneline 能看到两次提交历史

练习 10：再加一个文件

echo "pipeline practice notes" > notes.txt
git status
git add notes.txt
git commit -m "add notes"
git log --oneline

第 3 周的关键理解

你不用现在就掌握 Git 的全貌，但至少要先形成这个印象：

Git 记录的不是“最终文件夹长什么样”，而是“一连串可追溯的快照”。

Missing Semester 的 Git 讲义就是从这个模型出发讲的：commit 是快照，历史不是简单列表，而是 DAG；先有这个理解，后面学分支和合并会轻松很多。

第 1 周：重定向、管道、基础文本处理

这一周只练一个核心能力：

练习 1：先看 > 和 >>

创建文件：

echo "apple" > fruits.txt
cat fruits.txt

cat fruits.txt

echo "orange" >> fruits.txt
cat fruits.txt

这里你要体会两件事：
	•	> 是写入，如果文件已经存在，会覆盖原内容
	•	>> 是追加，在原内容后面继续写

练习 2：刻意感受覆盖

现在你应该只能看到一行。
这一步一定要亲手做，因为很多人后面写日志或结果文件时，就是在这里误覆盖了旧内容。Bash 手册明确说 > 对已存在文件会截断到 0 大小。

这一组是非常典型的“命令行文本流处理”起点。Missing Semester 也把这类 shell tools 视为高频命令行工作流的一部分。

第 2 周：最小 shell 脚本

这一周的目标不是学 Bash 全部语法，而是先建立一个感觉：

脚本就是把你本来要手敲的一串命令，保存成可重复执行的文件。

Missing Semester 明确说，shell 脚本适合 shell 相关任务，比如创建 command pipelines、把结果写入文件、从标准输入读取数据。

练习 6：你的第一个脚本

先造测试日志：

echo "error: login failed" > app.log
echo "info: job started" >> app.log
echo "error: file missing" >> app.log
echo "info: job finished" >> app.log

新建脚本：

touch count_errors.sh

把下面内容写进去：

#!/bin/bash
cat app.log | grep "error" | wc -l > error_count.txt
cat error_count.txt

运行：

bash count_errors.sh

如果能输出 2，说明你已经完成了一个最小可重复脚本。

练习 7：再做一个标签去重脚本

先造数据：

echo "ok" > labels.txt
echo "blur" >> labels.txt
echo "ok" >> labels.txt
echo "bad" >> labels.txt
echo "blur" >> labels.txt
echo "ok" >> labels.txt

新建脚本：

touch count_labels.sh

写入：

#!/bin/bash
sort labels.txt | uniq > unique_labels.txt
cat unique_labels.txt

运行：

bash count_labels.sh

练习 8：给脚本传最简单参数

新建脚本：

touch show_args.sh

写入：

#!/bin/bash
echo "script name: $0"
echo "first arg: $1"
echo "second arg: $2"
echo "all args: $@"
echo "arg count: $#"

运行：

bash show_args.sh aaa bbb ccc

Missing Semester 对 Bash 特殊变量的介绍里，明确列出了 $0、$1 到 $9、$@、$# 等参数变量。 

练习 9：练最简单条件判断

新建：

touch check_file.sh

写入：

#!/bin/bash

if grep "error" app.log > /dev/null 2> /dev/null
then
  echo "has error"
else
  echo "no error"
fi

运行：

bash check_file.sh

Missing Semester 也专门展示了用 grep ... > /dev/null 2> /dev/null 配合退出状态做判断的写法。

第 3 周：Git 最小工作流

这一周你只建立一个非常小但完整的 Git 闭环：

初始化仓库 → 看状态 → 暂存 → 提交 → 看历史

Missing Semester 解释了 Git 的核心模型：Git 把项目历史看成一系列快照，提交历史是 DAG；同时它也列出了最基本的一组命令：git init、git status、git add、git commit、git log。Pro Git 说明如果你要在已有目录里开始版本管理，就进入目录后执行 git init；如果要从现有仓库开始，则用 git clone。 

第 3 周先做 Git 基础配置

GitHub Docs 建议先下载并安装 Git，然后设置用户名和提交邮箱。对应命令可以这样做：

git config --global user.name "你的名字"
git config --global user.email "你的邮箱"

GitHub Docs 也给出了查看配置值的方式。 

你可以顺手确认一下：

git config --global user.name
git config --global user.email

练习 14：感受 git clone 是什么

这一步你不一定要真的去克隆大型仓库，但要先知道：
	•	git init 是在本地已有目录里开始版本管理
	•	git clone <url> 是拿到一个现有仓库的完整本地副本

Pro Git 明确说 git clone 默认会把项目完整历史拉下来，而不仅仅是工作区。

这一轮最容易踩的坑

第一个坑，是把 > 当成“总是安全的写文件”。
它会覆盖原文件内容，不是追加。Bash 手册对这一点写得很明确。 

第二个坑，是只会手敲命令，不会把命令存成脚本。
Missing Semester 明确把 shell scripting 视为“下一步复杂度”，因为它让命令链可以重复执行。 

第三个坑，是太早陷进 Git 的高级概念。
你现在最重要的是先理解 Git 记录的是快照历史，先把最小工作流跑通，而不是一上来就学 rebase、冲突解决和复杂分支模型。Missing Semester 也是先从快照和 DAG 讲起的。
